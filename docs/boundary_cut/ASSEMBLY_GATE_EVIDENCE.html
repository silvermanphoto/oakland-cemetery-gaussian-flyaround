VERDICT: FAIL — re-express the outline before cutting

The keep outline (Joel's hand-traced boundary) was turned into map coordinates using one photo set — the 300-foot oblique flight. The 3D tiles now waiting to be cut are built in a different, independently-computed coordinate system: the combined 300-foot + 125-foot solve. Each system is, on its own, accurately pinned to GPS (both within ~0.2 m of the drone's recorded positions) — but they are pinned independently, so relative to each other they sit at a slight angle and scale (~1.22° rotation, 0.66% scale, plus a few-metre shift).

Carried across that gap, the outline's 13 corners land 5.9–21.2 metres out of place. The recorded gate expected sub-metre agreement; the reality is up to a car-length-and-a-half of error, which would delete real cemetery on the north-east side and keep non-cemetery ground elsewhere.

The fix is prepared: boundary_keep_polygon_FINAL_c2.json is the same outline re-expressed in the tiles' own coordinate system — every corner in the right place, same acreage and shape. Offered for the lead to bless; not self-approved.

Independent confirmation

To rule out an artefact, the two solves' shared 3,270 camera positions were aligned directly to each other, using no GPS at all. That returns the same frame difference (scale 1.006591, rotation -1.2230°) and fits to a residual of about 0.03 m — the two coordinate systems are related by an almost-perfect rigid rotation-and-scale. Two independent methods agreeing to the centimetre make the corner errors trustworthy and the corrected _c2 outline exact.

Recommendation

The gate result is unambiguous. Cut the v2 tiles with boundary_keep_polygon_FINAL_c2.json, not the original. The correction is ready and independently checked; it awaits the lead's one-look nod (or a request for a visual overlay on a real tile) — it is not self-blessed.
