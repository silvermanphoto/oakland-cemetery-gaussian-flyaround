Blender Lane Plan

Oakland Cemetery Gaussian Flyaround (26-029) · verified 2026-07-21 · staging only, no GPU touched (training in flight)

Principle 5, literally  All 36 tiles in one Blender scene at full quality = not possible on Skychief — it would crash.

The 36 tiles are 135.7 million gaussians at full degree-3 color. That is ~32 GB of video memory for the raw data alone against the RTX 3090's 24 GB — the color coefficients by themselves (~24 GB) fill the whole card. In system RAM the add-ons expand it to 68–170 GB against 64 GB. And no Blender add-on is even built to hold that many (free ones: a few million; paid one's own manual: crashes past ~16 M). This is exactly the "machine literally crashes" case the principle says to bring to you with numbers.

Achievable  Full quality — no gaussian ever decimated — via per-shot region rendering.

The flyaround is a moving camera. Render it shot by shot: each shot loads only the full-count tiles its camera actually sees, renders those frames at full degree-3 color, frees them, moves on. Every rendered frame is full quality; off-screen or far tiles are simply not loaded (runtime level-of-detail — principle 3), which is not the permanent gaussian loss principle 5 forbids. The canary measures how many tiles fit per shot so the shot list is built from real numbers.

Recommendation

Canary KIRI 3DGS Render first (free), render the flyaround per-shot. KIRI is the only candidate that renders splats through real Cycles — genuine path-traced light, depth of field, motion blur, compositing — at full degree-3 color. It is free and reputable, so the canary costs nothing but a GPU gap. Its weakness is scale (each gaussian becomes a mesh face, happiest ~2–3 M), which the canary measures and which per-shot rendering accommodates.

Paid options — surfaced for your call, you buy (never me)

  • SplatForge — $49 one-time. Worth it if the KIRI canary proves KIRI too heavy per tile and you want a fast, faithful splat-look flyaround rather than a path-traced one. Full degree-3 color, highest documented ceiling (~16 M/scene), built-in animation export — but no in-Blender lighting/DoF/motion-blur (add in comp). Buy only after the free canary, and verify its ceiling on our data first (its store says "100 M", its own manual says "16 M then crash" — trust the manual).
  • Cloud GPU (48 GB A6000, ~$0.33/hr). Raises the memory wall so bigger shots fit, but not the add-on wall — it does not, by itself, buy the literal all-36 render. For "one seamless world" the right tool is the Unreal/NanoGS path you already have; Blender's job is the beautiful rendered reel.

Install target: Blender 4.5 LTS (free, blender.org; supported to 2027; covers all three add-ons). Skychief has only 3.0 and 3.4 today — an install is required (listed, not done).

All-36 = ~32 GB VRAM (over 24 GB) and 68–170 GB RAM in the add-on (over 64 GB). A shot's worth of tiles fits comfortably; the real per-shot count is add-on-bound and is what the canary measures. Linked collections help organization but Blender does not page them out of core — the lazy mechanism is load-subset → render → purge → next shot.

The canary (staged, not run)

Written and staged on Skychief at C:\blender_lane\ (mirrored in this repo at docs/blender_lane/). It self-guards on a GPU gap — it will not run while training is in flight (right now it would correctly abort: only 4.7 GB VRAM free).

  • run_canary.ps1 — GPU-gap guard, Blender locator (errors to the install step if only 3.0/3.4 present), a 2-second RAM+VRAM sampler for true peaks, and a hard timeout-kill so a runaway import ends as a clean killed process + full log (the crash-recovery path — crashing on a too-big import is an expected canary outcome).
  • canary.py — installs+enables KIRI headless, logs the add-on's real operator names, imports one tile, frames an oblique aerial camera (the "start inside the sweet spot" rule for an aerial capture), renders one EEVEE + one Cycles frame, writes counts + timings.

Map the curve across three tiles in one gap: tile_0_5 (0.19 M), tile_3_2 (4.3 M, obelisk cell), tile_1_4 (7.90 M, the stress test). Tiles live on Skychief at H:\2026 Files\26-002 Oakland Cemetery Splat\deliver\ue_tiles\.

powershell -NoProfile -ExecutionPolicy Bypass -File C:\blender_lane\run_canary.ps1 `
    -Tile "H:\2026 Files\26-002 Oakland Cemetery Splat\deliver\ue_tiles\tile_1_4.ply" -Engine EEVEE

Canary passes when: the largest tile imports without crashing, peak RAM stays well under 64 GB and peak VRAM under ~22 GB, the viewport is interactive enough to camera-animate, and both an EEVEE and a Cycles frame render out looking like the cemetery. Those five numbers size the shot list.

Open risks

  • KIRI's real per-tile ceiling on 24 GB is unmeasured — the single biggest unknown, and why the canary exists. A 4–8 M tile may need points-proxy or Cycles-only-for-final.
  • SplatForge 16 M vs 100 M — primary-source-vs-marketing contradiction; verify on our data before paying $49.
  • BlendSplat is degree-2 — a view-dependent-color step-down on our degree-3 tiles; your call, not chosen silently.
  • Headless operators — these add-ons are GUI-first; the canary discovers and logs the real operator names and fails loudly, so we learn the true API on the first cheap run.
  • Blender 5.2 — if you'd rather stay on the current 5.2 LTS, KIRI only offers an "untested, likely buggy" 5.2 patch (also staged). 4.5 LTS or 5.1 avoids that.
